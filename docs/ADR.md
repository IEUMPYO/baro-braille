# Architecture Decision Records

## 철학

현장 검증 먼저. 백엔드/AI 없이 플로우 확인. Mock 쓰되 나중에 API로 갈아끼울 수 있게.

---

### ADR-001: Mock Data 우선 접근

**결정**: 백엔드/AI 모델 없이 Mock data로 전체 사용자 플로우를 먼저 구현

**이유**:

- 백엔드/AI 모델 연구가 진행 중인 상황
- 기관/관계자 데모를 위해 빠른 시연 필요
- UI/UX 검증을 먼저 진행하고 싶음
- 실제 API 구조를 미리 설계하고 Mock으로 시뮬레이션 가능

**트레이드오프**:

- ✓ 빠른 프로토타입 제작 및 현장 피드백 수집
- ✓ UI 플로우 검증에 집중 가능
- ✕ 실제 성능/에러 케이스 테스트 불가
- ✕ Mock → API 전환 시 리팩토링 필요

---

### ADR-002: Next.js App Router 선택

**결정**: Next.js 16 App Router 기반 React 애플리케이션

**이유**:

- 정적 빌드(`output: 'export'`)로 GitHub Pages 배포 가능
- 이후 클라우드 서버 전환 시 API Routes, SSR 활성화 용이
- React 19의 최신 기능 활용 (useOptimistic, Server Actions 등 이후 사용 가능)
- 학습 곡선이 낮고 빠른 개발 가능

**트레이드오프**:

- ✓ 정적/동적 배포 환경 모두 지원 가능
- ✓ 광범위한 커뮤니티 지원
- ✕ App Router는 비교적 최신 기술 (안정성 고려)
- ✕ 정적 빌드 시 API Routes 사용 불가 (현재 단계에서는 문제 없음)

---

### ADR-003: GitHub Pages 정적 배포 (단계적 전환)

**결정**: 현재는 GitHub Pages로 정적 배포, 이후 자체 클라우드 서버로 전환

**이유**:

- 초기 비용 제로 (무료 호스팅)
- CI/CD 자동화 (GitHub Actions)
- 빠른 배포 및 버전 관리
- 기관 데모용으로 충분한 성능

**트레이드오프**:

- ✓ 빠른 배포 및 무료 호스팅
- ✓ 버전 관리 용이
- ✕ 정적 파일만 배포 가능 (서버 로직 불가)
- ✕ 실시간 API 연동 불가 (Mock만 가능)
- → 이후 클라우드 서버 전환으로 해결 예정

---

### ADR-004: TypeScript 미도입 (JavaScript 우선)

**결정**: 현재는 JavaScript로 개발, TypeScript는 이후 도입 검토

**이유**:

- MVP 프로토타입 단계로 빠른 개발 우선
- 타입 안정성보다 기능 검증이 우선
- 컴포넌트 구조가 단순하여 타입 관리 부담 낮음

**트레이드오프**:

- ✓ 빠른 프로토타입 제작
- ✓ 러닝 커브 감소
- ✕ 타입 안정성 부족 (런타임 에러 가능성)
- ✕ IDE 자동완성 제한적
- → 프로젝트 확장 시 TypeScript 마이그레이션 고려

---

### ADR-005: Feature-based 아키텍처 (도메인 기반)

**결정**: Next.js/React 생태계 표준인 Feature-based 아키텍처 채택

**이유**:

- 도메인별로 코드 응집도 향상
- 각 기능이 명확히 분리됨 (upload/preview/conversion/result)
- 협업 시 충돌 최소화
- Next.js 커뮤니티 표준 패턴

**구조**:

```
components/
├── upload/
├── preview/
├── conversion/
└── result/
```

**트레이드오프**:

- ✓ 코드 찾기 쉬움
- ✓ 기능 추가/삭제 용이
- ✓ 확장성 우수
- ✕ 폴더 구조 복잡도 증가
- → 현재 규모에서 적합한 선택

---

### ADR-006: Vanilla CSS (Tailwind 미사용)

**결정**: CSS 변수 기반 Vanilla CSS 사용

**이유**:

- 프로젝트 규모가 작아 Tailwind 도입 오버엔지니어링
- 디자인 시스템이 명확하여 CSS 변수로 충분히 관리 가능
- 번들 크기 최소화 (Tailwind 없이 더 작은 CSS 파일)

**트레이드오프**:

- ✓ 번들 크기 최소화
- ✓ 커스텀 디자인 완전 제어
- ✕ 클래스명 네이밍 필요
- ✕ 반응형 작업 시 미디어 쿼리 직접 작성
- → 현재 규모에서는 적합한 선택

---

### ADR-007: 단순한 Mock → API 전환

**결정**: `lib/services.js`에 Mock 로직 구현, API 전환 시 리팩토링

**이유**:

- MVP 단계에서 환경 변수 관리 불필요
- Mock 로직이 단순하여 파일 하나로 관리 가능
- API 전환 시점에 구조 재설계

**트레이드오프**:

- ✓ 현재 복잡도 최소화
- ✓ 빠른 개발
- ✕ API 전환 시 리팩토링 필요
- → MVP 목표에 부합
